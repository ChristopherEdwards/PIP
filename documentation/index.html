Welcome to FIS PIP™ V0.1

What is PIP?

From Fidelity National Information Services (FIS), PIP is a mature and proven database application development and deployment platform now released under the terms of the GNU General Public License v3. A deployment of PIP on FIS GT.M™ on Linux on x86 hardware provides a complete open source stack with superb transaction processing throughput and unique functionality for extreme levels of business continuity.

PIP includes a complete database management system: a SQL engine; a compiler for PSL (Profile Scripting Language), a lightweight, object-oriented scripting language; a JDBC driver; MTM, a lightweight communications monitor; and two modern graphical IDEs, one based on Eclipse and the other browser-based and driven by Tomcat on the server. Applications developed with PIP can be deployed on GT.M or on Oracle. PIP is designed so that applications created with PIP can in principle be deployed on any implementation of M or any implementation of SQL. At this time, GT.M and Oracle are the only deployment platforms on which PIP is tested and supported.

PIP is intended to be used to:

  • Develop transaction processing database applications that require:

    • robust support for ACID (Atomic, Consistent, Isolated, Durable) transactions,

    • performance & scalability to meet the throughput needs of the largest enterprise scale applications on a complete open source software stack, and

    • 24x365 continuity of business, in the face of not only unplanned events such as system outages but also in face of planned events such as system and application upgrades, even upgrades requiring a change to database schema.

  • Incrementally migrate applications written in M to relational schemas (where the underlying data relationships are relational) and object oriented business logic. An incremental migration can include:

    • mapping data elements and computed values to relational tables for reporting and decision support access via SQL/JDBC, and

    • incrementally migrating business logic from M to a combination of PSL, SQL and C through a process of evolution, while

    • retaining the ability to deploy applications with identical application business logic with the operational behavior of either an underlying M engine such as GT.M with its scalability, price/performance and open source licensing, or on a brand-name relational engine.

PIP provides a competitive relational / object oriented application development and deployment capability. What makes PIP exceptional is its level of control on the production deployment that allows it to provide the extreme levels of continuity of business, transaction throughput and cost-effectiveness. PIP also has an excellent browser based capability for ad hoc reporting, e.g., for use by business analysts for exploratory data analysis.

PIP is the infrastructure for the powerful FIS Profile® real-time core processing system for banks. PIP is mature because it has been in production for many years, but until now it has only been available as part of Profile – the code bases were intertwined. Now, for the first time, the PIP code base has been separated from the Profile code base and is separately released GPL software. The Profile code base remains proprietary.

This first release of PIP is packaged as a complete x86 virtual machine that boots and runs Debian GNU/Linux 4.0 r2. In the not-too-distant future, it is anticipated that PIP will be released as a stand-alone package.

Free community support for PIP is available on the forums of the PIP project at Source Forge. FIS provides support for PIP on a commercial basis. Please contact K.S. Bhaskar (ks dot bhaskar at fnis dot com) for more information.

Caveat

Although the underlying PIP engine is mature (it is in its third major version), the separation from Profile and stand-alone packaging is brand new. Please see the discussion on PIP Limitations for more information.

How do I use the PIP virtual machine?

The PIP Linux virtual machine distribution is provided as a disk image that you can boot and run with the open source processor emulator QEMU on virtually all popular host platforms. When running this virtual machine on x86 or x86_64 host hardware, the QEMU accelerator can be used to speed up performance.

The password for both pip and root is PIPRocks.

Available Hosts

PIP can operate as a virtual machine on Windows, Linux, Mac as well as other hosts. See below for details.

Windows hosts: QEMU and the accelerator are included in the package. Setup instructions are as follows:

  1. Install the accelerator by executing the file Kqemu-1.3.0pre11-install.exe. While use of the accelerator is optional, its use will significantly speed up the virtual machine.

  2. Unzip the contents of the archive qemu-0.9.0-windows.zip to a location of your choice (e.g., “C:\Program Files\ qemu-0.9.0-windows”)

  3. Unzip the file PIP_V01.qcow from the archive PIP_V01.zip to the directory where you have installed QEMU (in this example, the qemu-0.9.0-windows sub-directory).

  4. Boot the virtual machine with the following shortcut executed in the qemu-0.9.0-windows subdirectory:

qemu -kernel-kqemu -L . -m 512 -redir tcp:2222::22 -redir tcp:8280::8180 -redir tcp:10100::10000 -redir tcp:61012::61012 PIP_V01.qcow

Linux hosts: QEMU for Linux is include in the package repositories of major Linux distributions,. The virtual machine also works with kvm, the Linux kernel based virtual machine, and should work with other forms of virtualization (although the disk image may need to be converted with the qemu-img program). Depending on the virtualization support in your CPU and BIOS, one of the following commands should boot the PIP virtual machine:

qemu -kernel-kqemu -m 512 -redir tcp:2222::22 -redir tcp:8280::8180 -redir tcp:10100::10000 -redir tcp:61012::61012 PIP_V01.qcow

kvm -m 512 -redir tcp:2222::22 -redir tcp:8280::8180 -redir tcp:10100::10000 -redir tcp:61012::61012 PIP_V01.qcow

Depending on your Linux distribution, hardware, and system configuration, you may need to load the kvm kernel module (with modprobe kvm) or to build and load your QEMU accelerator module.

QEMU for Mac OS X as well as QEMU for OpenSolaris are available on the Internet in order to run the PIP virtual machine on those platforms.

Accessing PIP

Once the PIP virtual machine is booted, QEMU provides a character-based console to the virtual machine. You can connect to the virtual machine from this console. QEMU virtual machines run as if they are on a protected local area network (LAN) segment behind a firewall and proxy server on your host machine. Outbound network access from a virtual machine is routed through a virtual gateway that, on that LAN segment appears at address 10.0.2.2. The command line used to start up a virtual machine specifies TCP ports on the host that are to be forwarded to ports on the virtual machine. For “well known” ports, the suggested command lines use different ports on the host and guest machines (e.g., since port 22 on the host is likely to be used for ssh sessions to the host, and is therefore an unsuitable port to forward connections to port 22 on the virtual machine, port 2222 on the host is used instead).

  • For greater flexibility (e.g., scrollable session log) and for additional character-based sessions, you can use a secure shell (ssh) client on your host, such as putty, to connect to the virtual machine by connecting to port 2222 on localhost (127.0.0.1). Use userid pip to login to the virtual machine.

  • Port 8280 on the host is forwarded to port 8180 on the virtual machine for Tomcat administration (connect with http://localhost:8280). Connect as user tomcat password tomcat.

  • Port 10100 on the host is fowarded to port 10000 on the virtual machine to use Webmin for administration of the virtual machine (connect with https://localhost:10100 and login as root).

  • Port 61012 on the host is forwarded to port 61012 on the virtual machine. This is the port on which PIP listens and from which the browser based IDE is provided (connect to http://localhost:8280/ProfileBrowserIDE/pages/ProfileBrowserIDE.html as user e0101001 password xxxx). Online documentation for PIP is available via this interface, and is a good place to begin exploring PIP.

  • Port 61012 is also used for the Eclipse based IDE. See instructions for Access from Eclipse for more information.

  • Port 61012 is also used for access to PIP by the JDBC driver.

Note: Although the virtual machine is character-based, QEMU can grab the mouse for the virtual machine if the mouse is clicked with the pointer inside the virtual machine window. Use the key combination Control-Alt to release the mouse and return control to the host machine.

Access from Eclipse

Eclipse 3.3 is the tested and recommended version of Eclipse. Eclipse 3.2 appears to work, potentially with some glitches.

  1. Install the PIP IDE in Eclipse: From Eclipse, choose Help / Software Updates / Find and Install. Choose “Search for new features to install” and click on Next. Click on “New Remote Site”, choose the URL http://localhost:8280/piptools and give it a name such as PIP Demo. This brings it into the list of update sites to visit. Choose PIP Demo and click on Finish. Check PIP Demo in the Search Results dialog box, and click on Next. This brings up a dialog box where you can accept the software license (GPL v3) and click Next, followed by Finish, and then Install All to install the Eclipse based PIP IDE.

  2. Create a new project: File / New / Project / New Profile Project and click on Next. Choose a name for your project, e.g., “My PIP” and click on Finish to create the project.

  3. Create a connection to the PIP environment in the virtual machine: Right click on the “My PIP” project and choose Properties. Click on the arrow to expand “FIS Profile”, click on “Profile Login” and click on Select. Click on Add and specify pipv01 as the Environment, localhost as the server, 61012 as the port, SCA$IBS as the IBS Server Name, 1 as the User ID and xxx as the Password. Click on Test to verify the connection.

You are now set to use the Eclipse IDE to explore and program PIP.

JDBC access

A JDBC driver is provided for access to PIP from other than Eclipse and browsers (e.g., from an office suite or a reporting tool), as well as user documentation for the driver. Whereas the JDBC driver has been edited to remove those parts of the software that pertain to Profile applications rather than to PIP, the JDBC user documentation has not yet been correspondingly edited.

Documentation

In addition to the JDBC documentation, there is online user documentation accessible via the browser IDE as well as with Eclipse. Since PIP is bootstrapped, information about PIP architecture is itself visible and can be manipulated through the IDEs. Caveat: this is a useful and powerful capability in a development environment. Use it with care, because you can potentially break PIP and lock yourself out of being able to repair the damage from an IDE. Repair actions would need to be taken by connecting to the PIP virtual machine via ssh and working with text files.

Shutting down the PIP virtual machine

To shut down the PIP virtual machine, connect to Webmin. Click on “System” to expand the list of system configuration options. Click on “Bootup and Shutdown” and scroll down to the bottom. There are buttons labelled “Shutdown System” and “Reboot System”.

Alternatively, login as root to the virtual machine on the console or via an ssh session, and execute the command: shutdown -h now

PIP Limitations

Although PIP's underlying engine is mature, the separation from Profile and stand-alone packaging is brand new. Bearing in mind the open source mantra of “release early, release often”, we are making PIP available for the community to gain experience with it even as we continue to refine it. Therefore:

  • Separation of PIP from Profile: There are known references to Profile, and potentially code and data that should belong with Profile that inadvertently remain with PIP. There are also likely cases where code and data that should belong with PIP inadvertently remain with Profile. For example, the table CUVAR includes computed columns that call code that belongs with the Profile application.

  • Configuration / Separation of PIP from Applications / Deployment: There is not yet a clean separation of PIP code and data from that of applications developed with PIP. Ideally, there would be one copy of PIP on a system, and multiple application development environments that share that one copy of PIP. At this time, a copy of PIP must exist for each separate application development environment. Also, when deploying applications developed with PIP, it is not desirable, and should not be necessary to deploy all of PIP. Today, it needs to be.

  • Documentation: Although there is much documentation for Profile, documentation of the infrastructure is inseparably entwined with documentation of the banking functionality. We are aware that we need to create new documentation for PIP. Please contact us if you would like to volunteer to help us with documentation.

  • Testing: Although Profile is extensively tested, the infrastructure is exercised by test cases that drive the banking functionality, and there is only limited stand-alone testing of the infrastructure (as a stand-alone product, GT.M is extensively tested in its own right). We are aware of the need to better test PIP when it is not embedded in banking functionality. Please contact us if you would like to volunteer to help us with automated testing.

  • Security: Although both Profile and GT.M have been extensively vetted as applications in their own right, PIP as a stand alone application has not. There is not yet a separate security model or policy document for PIP. Furthermore, since PIP is an application development platform, any security model for PIP is clearly different from the security model of an application developed with PIP and then deployed.

None of the above limitations is intended to dissuade you from developing and deploying applications with PIP. However, it does mean that despite the maturity of the underlying engine in PIP, developing and deploying applications with PIP today will take more expert knowledge than we would like it to.

Please report issues (bugs, misfeatures as well as suggestions for improvement) as you find them, and we will try to address them. We will give priority to changes with high impact (including closing security gaps), low effort, and to those items that have real value to real users (in the open source software business we must be extra attentive to the needs of those with checkbooks).

Related FAQs

What is FIS Profile?

FIS Profile is a tightly integrated real-time core processing system which offers financial institutions premier transaction engine, product factory, and information repository capabilities. Profile eliminates many of the functional, informational, and delivery boundaries that are traditionally found in many legacy banking enterprise systems.

A core processing system is the legal system of record for a bank, and its single most mission critical application. Built on FIS PIP, Profile runs the largest real time core processing system that is in daily production at any bank anywhere in the world that we know of, and has recently been successfully benchmarked on an x86_64 Linux platform on a database 50% larger than that largest live system.

FIS Profile represents a new high standard of market and customer responsiveness. Profile features a flexible electronic manufacturing model — an innovative architecture that enables the dynamic and efficient assembly of banking products, instruments, and services from an extensive inventory of business components. Thanks to its customer-centric design, the powerful Profile service architecture and data model places customers at the center of the information repository, thus enabling institutions to offer true relationship banking.

Profile is the system of record for many tens of millions of bank accounts around the world, and derives its flexibility, performance and functionality from the PIP infrastructure on which it is built.

What is GT.M?

GT.M is the core technology underlying PIP. It consists of a high performance associative memory data store (database) with semantics for full ACID (Atomic, Consistent, Isolated, Durable) transactions, and functionality to create unique logical multi-site application configurations for unparalleled continuity of business in the face of not only unplanned events (such as unexpected system and site outages) as well as planned events (such as system and application software upgrades, and even most upgrades that involve a database schema change). In addition to being the platform for most Profile deployments, GT.M is also increasingly used in other industries such as health care, and is entrusted with the bank balances and medical records of millions of people. GT.M also provides a true compiler for the scripting and application development language M (also known as MUMPS).

GT.M is mature, robust, scalable, secure, and has been in production use since 1986.
