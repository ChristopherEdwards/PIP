Issues to Review for Custom Code

If using the PSL code generator in Profile v6.4 and Profile v7.0, review the following items prior to writing custom code. These versions of PSL are not backward compatible with the following classes, methods, etc.

Record Class

The copy method previously accepted a parameter that would change the mode of the record being copied to. The copy method creates a second version of an instantiated record. In some cases, the new object will be filed into the database with a different set of keys. For example, escrow account creation copies the address record from the loan and saves it to the database using its own account number. The required change is to remove the parameter from the use of the method and add a second line of code that uses the record.setMode method.

Row Class

Row columns that are named the same as one in the dep object will be moved into that location. All others such as the XXX name must be manually defined in the column Map parameter.

Schema Class

The previous iteration of the createTable method allowed the use of literals as a key. However, the quotes that were required around the literals did not follow the industry standard of “”literal”” since the keys were embedded within quotes. This has been changed. The compiler now prints an error message when using the previous literal format in the schema create.

Db Class

  • The Db.isDefined method cannot be used with tables that contain only one record (e.g., Institution Variables table (CUVAR)), because the Db.isDefined method requires keys. The code needs to be written to use getRecord with a new third parameter, which does not throw an error if the record does not exist.

  • The Db.isDefined method previously allowed the programmer to not quote the keys. This error has been closed in this generation.

  • Previously, the compiler ignored PSL code that contained an extra } or ). If the PSL code contains an extra } or ), the compiler now reports an error during the compilation process. This occurs in Db.select and Db.getOneRow coding.

  • The getOneRow method was rewritten to provide a better review of the keys supplied for the method. If an incorrect number of keys was supplied before the change, the compiler did not recognize it and the data retrieved was null. Due to the rewrite, the compiler will throw an error if the method receives the wrong number of keys to properly retrieve data. This method also previously accepted the word ‘distinct’ as a column, but did not compile the code correctly without warning. This situation will now throw an error.

Issues Unrelated to a Class

  • Using $Get around a PSL object was previously allowed, but the compiler threw it out. It is now reported as an error.

  • PSL cannot allow exclusive new commands. The effect of their use is that the internal object array is hidden. Even though the object pointer could be in the exclusive list, the objects would not be. The compiler now throws an error when an exclusive new is found in the code.
