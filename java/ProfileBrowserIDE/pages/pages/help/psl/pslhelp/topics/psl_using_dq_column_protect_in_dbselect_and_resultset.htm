Using DATA-QWIK Column Protection in Result Sets

PSL supports the DATA-QWIK column protection scheme, provided you adhere to the following rules.

  • The protection scheme relies on passing a literal PSQLparameter containing "/PROTECTION=n" to the Db.select() method that instantiates the ResultSet. The PSQLparameter is the sixth parameter in the Db.select() method. The values of the /PROTECTION parameter correspond to the values of the Data Item Protection Logic attribute (DBTBL5H.RESFLG).

  • If "/PROTECTION=0" is specified, then the generated code ignores the data item protection schema. Both ResultSet.getColProt() and ResultSet.getRowProt() will return an empty string.

  • If "/PROTECTION=1", then the generated routine includes code that maintains the protection values of the individual columns, if at least one table specified in the Db.select() method has a data item for which protection is defined. Because the PSL compiler must be able to detect this at compile time, the tableName parameter of the Db.select() method must also be literal.

    Because the intent of "/PROTECTION=1" is to disallow access to individual column values, but allow column usage in aggregates, the developer must look at and interpret the individual column protection values through ResultSet.getColProt().

  • "/PROTECTION=2" implies that neither access to individual column values nor access through aggregates is allowed when the column is protected. In this case, ResultSet.next() will replace the column value by NULL when access to the column is not granted. Because ResultSet.next() forces the NULL, both ResultSet.getCol() and ResultSet.getRow() will return NULL as the column value.
